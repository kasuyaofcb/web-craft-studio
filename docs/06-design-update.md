# 6. デザイン変更

> **このフェーズで**: サイト完成後、デザインカンプの変更に対応してコーディングを行います。

## 📌 このフェーズの目的

- 完成したサイトのデザインを変更する
- Figmaで修正されたデザインカンプを確認
- 修正されたデザインをコーディングで実装

## 🔧 作業の流れ

1. **デザインカンプの確認**（Figmaで修正されたデザインを確認）
2. **変更箇所の特定**（どのセクション・要素が変更されたか）
3. **ブランチの作成**（Issueにアサインされたらすぐ）
4. **変更前の状態を確認**（スクリーンショットを撮影）
5. **コーディング**（HTML/CSS/JavaScriptの修正）
6. **動作確認**
7. **コミット・PR作成**

## 📝 作業手順

### 1. デザインカンプの確認（Figma）

1. **Figmaファイルを開く**
   - チームで共有されているFigmaファイルにアクセス
   - 最新のデザインカンプを確認

2. **変更箇所の特定**
   - どのセクションが変更されたか確認
   - 変更前と変更後を比較
   - 色、フォント、余白、レイアウトなどの変更点をメモ

3. **デザインの詳細確認**
   - 変更された要素のサイズ、色、フォントサイズを確認
   - レスポンシブ対応（PC/タブレット/スマホ）を確認
   - インタラクティブな要素（ホバー、クリックなど）を確認

### 2. 変更箇所の特定

1. **どのファイルを修正するか確認**
   - 変更されたセクションに対応するファイルを特定
   - 例: FVセクションが変更された場合 → `section-fv.scss` と対応するテンプレートファイル

2. **変更前の状態を記録**
   - ブラウザで現在の状態を確認
   - スクリーンショットを撮影（変更前後の比較用）

### 3. ブランチの作成

Issueにアサインされたら、すぐにブランチを作成します：

```bash
# 最新のmainブランチを取得
git checkout main
git pull origin main

# 新しいブランチを作成（例: Issue #50 でFVセクションのデザイン変更）
git checkout -b issue-50-fv-design-update
```

> **💡 ブランチ名の例**:
> - `issue-50-fv-design-update` - FVセクションのデザイン変更
> - `issue-51-header-color-change` - ヘッダーの色変更
> - `issue-52-services-layout-update` - サービスセクションのレイアウト変更

### 4. コーディング

#### HTMLの修正（必要に応じて）

構造が変更された場合は、HTMLも修正します：

```html
<!-- 変更前 -->
<section class="sectionFv">
  <h1 class="sectionFv__title">タイトル</h1>
</section>

<!-- 変更後（例: サブタイトルが追加された場合） -->
<section class="sectionFv">
  <h1 class="sectionFv__title">タイトル</h1>
  <p class="sectionFv__subtitle">サブタイトル</p>
</section>
```

#### SCSSの修正

1. **対応するSCSSファイルを開く**
   - 例: FVセクションの場合 → `assets/scss/top/section-fv.scss`（WordPress化後）

2. **デザインカンプに合わせてスタイルを修正**
   ```scss
   .sectionFv {
     // 変更前
     // padding: 8rem 0;

     // 変更後（Figmaで確認した値に合わせる）
     padding: 10rem 0;

     // 背景色が変更された場合
     background-color: variables.$color-primary;
   }

   .sectionFv__title {
     // フォントサイズが変更された場合
     font-size: 4rem; // Figmaで確認した値

     // 色が変更された場合
     color: variables.$color-text;
   }
   ```

3. **レスポンシブ対応**
   ```scss
   .sectionFv {
     padding: 10rem 0;

     @include mixins.mq('md') {
       padding: 8rem 0; // タブレットサイズ
     }

     @include mixins.mq('sm') {
       padding: 4rem 0; // スマホサイズ
     }
   }
   ```

#### JavaScriptの修正（必要に応じて）

インタラクティブな要素が変更された場合は、JavaScriptも修正します：

```javascript
// 変更前の動作を確認
// 変更後の動作を実装
```

### 5. 動作確認

1. **ブラウザで確認**
   - Local by Flywheelでサイトを表示
   - 変更したセクションがデザインカンプ通りになっているか確認
   - PC/タブレット/スマホの各サイズで確認

2. **デザインカンプとの比較**
   - Figmaのデザインカンプと実装を比較
   - 色、フォントサイズ、余白などが一致しているか確認

3. **他のページへの影響を確認**
   - 変更したスタイルが他のページに影響していないか確認
   - 特に共通コンポーネント（header, footerなど）を変更した場合

### 6. コミット・PR作成

```bash
git add .
git commit -m "feat: FVセクションのデザイン変更

- パディングを8remから10remに変更
- タイトルフォントサイズを3.6remから4remに変更
- 背景色を変更
- レスポンシブ対応を更新"
```

その後、GitHubでプルリクエストを作成します。

## 💡 ポイント

- **デザインカンプをよく確認**: Figmaで正確な値を確認してからコーディング
- **変更前の状態を記録**: スクリーンショットを撮っておくと比較しやすい
- **小さな変更ごとに確認**: 一度に全部変更せず、小さな変更ごとに動作確認
- **レスポンシブ対応を忘れずに**: PCだけでなく、タブレット・スマホサイズも確認

## 🆘 よくある問題

### デザインカンプと実装が合わない

**確認事項**:
1. Figmaで正確な値を確認しているか（px単位など）
2. フォントサイズ、余白、色などが正しく実装されているか
3. レスポンシブ対応が正しく実装されているか

**解決方法**:
1. Figmaのデザインカンプを再度確認
2. ブラウザの開発者ツールで実際の値を確認
3. 必要に応じて、デザイナーに確認

### 変更したスタイルが他のページに影響している

**確認事項**:
1. 変更したクラス名が他のページでも使用されていないか
2. 共通コンポーネント（header, footerなど）を変更した場合、影響範囲を確認

**解決方法**:
1. 変更したクラス名を検索して、使用箇所を確認
2. 必要に応じて、新しいクラス名を作成
3. 影響範囲を確認してから変更

詳細は [よくある質問・トラブルシューティング](faq-troubleshooting.md) を参照してください。
